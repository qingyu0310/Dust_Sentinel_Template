# HPM UART 的 TRGM/GPTMR RX Idle 实现说明

面向对象：硬件工程师  
目的：说明 HPM 平台上，为什么会存在 `UART RX + DMA + TRGM + GPTMR` 这条软 idle 检测链路，它在硬件上需要什么条件，以及 `HPM5361` 当前板子为什么不成立。

## 1. 先说结论

`TRGM/GPTMR` 这条链路不是“软件写上去就能用”的功能，它本质上依赖：

1. `UART RX` 这根真实外部信号线
2. 同时进入一个 `TRGMx_Pn` 输入脚
3. `TRGM` 再把这个输入分发给 `GPTMR`
4. `GPTMR` 用捕获/比较来判断“开始收”和“已经 idle”

所以这条链路的硬件前提非常明确：

**UART 接收线，必须被 `TRGM` 看见。**

如果 `TRGM` 没看见这根线，软件层再怎么配置 `TRGM/GPTMR` 也不会触发。

## 2. 这条链路是怎么工作的

目标不是“收数据”，而是“判断一段 UART 数据什么时候结束”。

数据接收本身仍然是：

- UART 外设收字节
- DMA 把字节搬进内存

而 `TRGM/GPTMR` 做的是另一个任务：

- 监视 `RX` 线上有没有边沿变化
- 一旦有下降沿/开始位，认为“开始接收”
- 再启动一个定时器
- 如果在设定时间内没有新的位边沿，认为“这一帧结束了”
- 然后软件去 flush DMA，往上层发 `UART_RX_RDY`

可以把它理解成：

`UART 负责收字节`  
`DMA 负责搬字节`  
`TRGM/GPTMR 负责判断这一批字节什么时候算一帧结束`

## 3. 为什么要用 TRGM/GPTMR

不是所有平台都直接用 UART 自己的 `RX idle` 硬件标志。

有些平台或老实现会用：

- `TRGM`：把某根外部输入信号分发到内部模块
- `GPTMR`：测量这个信号多久没变化

这样就能在没有直接 UART idle 标志，或者不使用 UART 自带 idle 的实现里，自己拼一条“软 idle”检测链。

## 4. 硬件上需要满足什么

### 4.1 必须有一根 TRGM 输入脚

软件里常见的配置是这种意思：

- `uart-idle-trgm-input-src = <TRGM0_P5>`
- `uart-idle-trgm-output-gptmr-in = <GPTMR3_IN2>`
- `uart-idle-trgm-output-gptmr-sync = <GPTMR3_SYNCI>`

它翻成硬件话就是：

- 先让 `TRGM0_P5` 这个输入脚接收到 UART RX 的电平变化
- 再把它送给 `GPTMR3`

如果 `TRGM0_P5` 没有实际输入，后面的 `GPTMR` 永远不会动。

### 4.2 UART RX 和 TRGM 输入必须同时成立

这一步最容易误解。

很多芯片一个 PAD 可以复用成：

- `UART4_RXD`
- `TRGM0_P_05`

但这不代表它能**同时**作为这两种功能使用。

通常它的意思只是：

**这一个 PAD 可以二选一地配置成 UART4_RXD 或 TRGM0_P_05。**

不是“一个 PAD 同时干两件事”。

## 5. HPM5361 这块板为什么现在不成立

### 5.1 当前已知条件

当前板上已经确认：

- `UART4_RX` 只接到了 `PA17`

而官方 `HPM5361` 定义里：

- `PA17` 可以是 `UART4_RXD`
- `PA17` 也可以是 `TRGM0_P_05`

但这两者是不同的复用功能，不是同时生效。

### 5.2 这意味着什么

如果 `PA17` 被配置成：

- `UART4_RXD`

那它就不是：

- `TRGM0_P_05`

于是结果就是：

- UART4 的确能收到字节
- DMA 的确能搬数据
- 但 `TRGM0_P5` 这条输入链根本没接通
- `GPTMR` 看不到任何边沿变化

所以 `TRGM/GPTMR` 这条软 idle 链在当前接法下不成立。

## 6. 对硬件来说，这条链路要怎么做才成立

要让 `TRGM/GPTMR` 真正可用，硬件要满足下面任一方案。

### 方案 A：UART RX 物理上并到一个专用 TRGM 输入脚

也就是同一根外部信号线同时去：

- `UART4_RXD`
- `TRGM0_Pn`

这样：

- UART 能收数据
- TRGM 也能看到边沿

这是最直观、也最可靠的硬件方案。

### 方案 B：芯片/封装/内部矩阵本身支持同一 PAD 同时被 UART 和 TRGM 观察

如果硬件手册明确写了：

- 某个 PAD 在作为 UART RX 时，内部仍可被 TRGM 旁路观察

那才可以只接一根线。

但对当前 `HPM5361 + PA17` 这块板的已知情况，**不能默认这样成立**。

## 7. 现在硬件该重点确认什么

### 7.1 先确认设计目标

硬件要先明确自己到底想要哪条方案：

1. 走 `UART 自带 RX idle`
2. 走 `TRGM/GPTMR 软 idle`

这两条路的硬件要求不同。

### 7.2 如果坚持走 TRGM/GPTMR

请硬件确认：

1. `UART RX` 这根线有没有同时进到某个 `TRGM0_Pn`
2. 这个 `TRGM0_Pn` 对应的是哪一个真实 PAD
3. 这个 PAD 是否真的在 PCB 上有连接
4. 不是只在 `IOMUX` 头文件里“支持这个功能”，而是板子上真的接了

### 7.3 如果没有专用 TRGM 输入

那就不要再要求软件走 `TRGM/GPTMR` 这条路。

这种情况下更合理的是：

- 直接使用 `UART 硬件 RX idle`

也就是让 UART 自己判断一帧结束，而不是靠 `TRGM/GPTMR` 拼外部检测链。

## 8. 对 HPM5361 当前板子的建议

基于当前已知信息：

- `UART4_RX` 只在 `PA17`
- 没有确认到第二根独立 `TRGM0_Pn` 输入

所以建议是：

### 软件方案建议

优先走：

- `UART 硬件 RX idle`

不要把 `TRGM/GPTMR` 当成当前板子的主方案。

### 硬件方案建议

如果后续一定要支持 `TRGM/GPTMR`：

- 下一版原理图/PCB 把 `UART RX` 同时引到一个明确的 `TRGM0_Pn` 输入脚

不要依赖“同一个 PAD 名义上既支持 UART 又支持 TRGM”这种前提。

## 9. 给硬件的最简判断标准

只看一句话：

**TRGM/GPTMR 要成立，TRGM 输入脚必须真的收到 UART RX 的电平变化。**

如果做不到这一点，这条链路在硬件上就是不成立。

## 10. 当前问题的本质

当前 `HPM5361` 这块板的问题不是：

- DMA 不会搬
- UART 不会收
- 协议层不会解

而是：

**用 `TRGM/GPTMR` 去判断 UART 帧结束这件事，在当前硬件接法下没有成立条件。**

所以如果软件表现为：

- 收得到数据
- 但不能按 idle 正常切帧

这与当前硬件条件是完全一致的。
