# 三个嵌入式框架对比总结

## 对比对象

本文对比三个我当前代码演进阶段中具有代表性的项目：

1. `temp/basic_framework-master`
2. `temp/Dust_SentinelRobot_L_Game`
3. 当前仓库 `tflm`

它们不是简单的“谁更高级”，而是分别代表了不同阶段、不同目标下的架构选择。

---

## 一句话结论

- `basic_framework-master` 最像一套长期服役的通用基础平台，规则最完整，传承性最强。
- `Dust_SentinelRobot_L_Game` 最像围绕一台具体机器人快速打通的实战工程，业务闭环最强。
- 当前 `tflm` 最像在前两者基础上重新抽象出来的新一代框架，分层最清晰，未来延展性最好。

如果把三者放到一条演进线上看，可以理解为：

`basic_framework -> Dust_SentinelRobot_L_Game -> tflm`

前者提供“规则化框架视角”，中间提供“整机落地经验”，后者则开始形成“自己的架构语言”。

---

## 一、basic_framework-master

### 定位

这是一个偏“队伍级通用底座”的框架，目标不是只服务某一台机器人，而是服务多个机器人、多个成员、多个赛季的复用和传承。

### 优点

#### 1. 规则最完整

它明确提出了 `bsp / module / app` 三层结构，并且不是只停留在目录命名，而是把设计思想、任务频率、数据流、初始化流程、单双板切换策略都写进了文档。

这意味着它最强的地方不只是代码，而是“制度”。

#### 2. 文档体系最成熟

它的 README、`.Doc`、各层说明文档、模块说明文档都很齐全，明显是面向“新人可上手、队伍可传承”去设计的。

这一点非常重要，因为很多嵌入式项目代码能跑，但没人能接。

#### 3. 应用层解耦意识很强

它在 app 层主动引入 `message_center` 这种 pub-sub 机制，强调 `robot_cmd / chassis / gimbal / shoot` 之间不要直接相互包含，而要通过消息交换。

对于 C 风格 STM32 工程来说，这已经是很明显的架构意识了。

#### 4. 多机器人、多板卡复用能力强

它通过 `robot_def.h` 做单双板、兵种、参数切换，这种方式虽然不算最先进，但很适合“一个队伍多台车共用一套底座”。

### 缺点

#### 1. 规则强，但仍偏手工约束

很多边界是靠文档、命名和开发纪律维持，而不是靠语言机制或更强的编译期约束维持。

例如：

- 大量宏配置
- 手工注册 topic
- 手工安排任务频率
- 手工维护参数和外设归属

这使它很强，但也容易随着规模增长出现“规则知道，但执行走样”的问题。

#### 2. 与 STM32 HAL / CubeMX 耦合较深

它本质上还是一套基于 STM32F4 + CubeMX + FreeRTOS 的平台，移植性主要体现在“同类单片机工程”内，而不是更高层面的平台无关。

#### 3. pub-sub 已经有了，但机制本身还比较轻

`message_center` 更像一个工程内消息中间件，而不是非常严格的数据总线机制。

优点是轻、直接、够用；缺点是当系统继续膨胀时，数据一致性、队列语义、并发保护可能会成为进一步演进的瓶颈。

### 适合的场景

- 队伍传承
- 多机器人共底座
- 给新人提供一套能直接上手的工程平台
- 以 STM32 裸机/HAL/FreeRTOS 为主的软件栈

## 二、Dust_SentinelRobot_L_Game

### 定位

这是一个明显偏“具体机器人落地工程”的项目。

它虽然也有框架意识，但核心目标不是打造一套长期通用 SDK，而是围绕一台哨兵机器人，把底盘、云台、装填、裁判系统、上下板通信等全部打通。

### 优点

#### 1. 业务闭环最强

这个项目最大的优点是“真能打”。

从启动链路到总控逻辑，再到底盘、云台、通信、裁判系统、超级电容，已经是完整的整机工程，而不是只把底层模块封装一下。

#### 2. Robot 总控非常直观

它通过 `Robot` 作为整机抽象，把：

- `McuComm`
- `Chassis`
- `Gimbal`
- `Reload`
- `Imu`
- `Supercap`
- `Referee`

这些核心对象都装在一起，再用 `Robot::Task()` 完成跨模块决策。

这种设计虽然不是最解耦，但在整机项目里非常直观、非常高效。

#### 3. 控制逻辑已经不是“玩具级”

像底盘部分已经有：

- 工作模式切换
- 坐标变换
- 斜坡规划
- 运动学逆解
- 功率控制

云台部分也有：

- 角度环
- 速度环
- 滤波
- 电机失联恢复

这说明它不只是“会调 PID”，而是已经在组织完整控制链了。

#### 4. 落地经验浓

它的代码里有很多工程味很重的东西：

- CAN 回调直接分发到具体对象
- 任务内临界区复制共享数据
- 外设和业务对象强绑定
- 失联保护和运行模式切换

这些都很像真实机器人项目，而不是实验性质工程。

### 缺点

#### 1. 总控强，但总控偏重

`Robot` 很像“大脑”，很多跨模块策略都堆在里面。

短期看这很好，因为逻辑集中、调试方便；长期看容易变成“超大总控文件”，后续拆分难度会上升。

#### 2. 边界更多靠作者脑内维持

它有结构，但没有像 `basic_framework` 那样把规则制度化得特别清楚。

所以它更像“优秀的具体项目”，而不是“可广泛继承的平台”。

#### 3. 复用性不如另两者

里面很多设计都围绕当前这台哨兵机器人的设备拓扑和任务关系展开，迁移到其他机器人时，需要作者自己做较多判断和裁剪。

### 适合的场景

- 某一台具体机器人快速迭代
- 团队中有较强主导者，能持续维持整体结构
- 追求整机联调效率和实战闭环，而不是先做平台化

## 三、当前 tflm

### 定位

当前 `tflm` 更像是对前两个阶段经验的“再抽象”。

它已经不满足于“把机器人做出来”，而是在主动思考：

- 哪些是项目层
- 哪些是驱动层
- 哪些是模块层
- 哪些是算法层
- 哪些是消息层
- 哪些是板级配置

这说明它的目标更偏向“可迁移框架”。

### 优点

#### 1. 分层最清晰

当前仓库把结构拆成了：

- `projects`
- `drivers`
- `algorithm`
- `modules`
- `topic`

这比前两个 STM32 工程更干净，也更接近真正意义上的框架设计。

#### 2. 项目层与框架层分离得更主动

`README` 已经明确提出：

- `projects/` 是用户代码
- `apps / boards / thread` 是项目单元
- 其余部分是集成环境

这是一个非常重要的抽象提升。

#### 3. 启动流程更像“装配系统”

`main -> System_Bsp_Init -> System_Modules_Init -> System_Thread_Start`

这种写法已经从“项目写法”向“框架装配写法”进了一步。

#### 4. 构建裁剪意识最强

当前 `tflm` 已经把：

- `Kconfig`
- `CMake`
- feature toggle
- topic 编译条件
- thread 编译条件

结合起来了。

这意味着它不只是“分文件夹”，而是真的在做裁剪型架构。

#### 5. 更容易走向跨板卡、跨项目迁移

尤其 `boards + thread + apps` 的组合思想，已经明显比前两个项目更容易迁移和重组。

### 缺点

#### 1. 规则已经很先进，但实现还在收拢中

也就是说，现在 `tflm` 最大的问题不是架构想法不够，而是部分历史实现还没完全收敛到这套规则里。

例如线程入口、模块职责、项目装配点，已经有很清楚的设计，但实际代码中仍能看到过渡期痕迹。

#### 2. 当前更像“正在长大的框架”

相比 `basic_framework-master` 那种已经长期服役的稳定底座，`tflm` 还处于不断演化阶段。

这不是坏事，只是说明它的主要价值在“未来潜力”而不在“历史稳定性”。

#### 3. 学习门槛反而更高

因为它的抽象层次更高，新人如果没有 STM32/RTOS/模块化思维基础，未必比 `basic_framework-master` 更容易直接理解。

### 适合的场景

- 想沉淀自己的框架方法论
- 想做跨项目、跨板卡、跨模块复用
- 想让架构边界更清晰
- 未来还希望继续接入更多算法、推理、板级适配能力

---

## 四、三者横向对比

### 1. 从“平台化”能力看

- 最强：`basic_framework-master`
- 次强：`tflm`
- 最弱：`Dust_SentinelRobot_L_Game`

原因很简单：

- `basic_framework-master` 已经是面向队伍长期复用的平台
- `tflm` 的抽象比它更先进，但还在演进中
- `Dust_SentinelRobot_L_Game` 本质上还是具体机器人项目

### 2. 从“整机落地能力”看

- 最强：`Dust_SentinelRobot_L_Game`
- 次强：`basic_framework-master`
- 当前潜力大：`tflm`

`Dust_SentinelRobot_L_Game` 的业务链是最直接、最具体、最贴近实战联调的。

### 3. 从“架构清晰度”看

- 最强：`tflm`
- 次强：`basic_framework-master`
- 第三：`Dust_SentinelRobot_L_Game`

这里的“清晰度”是指概念边界是否干净，不是代码是否更复杂。

### 4. 从“文档与传承”看

- 最强：`basic_framework-master`
- 次强：`tflm`
- 第三：`Dust_SentinelRobot_L_Game`

`basic_framework-master` 明显是站在“让一个队伍持续使用”的角度去写文档的。

### 5. 从“未来扩展空间”看

- 最强：`tflm`
- 次强：`basic_framework-master`
- 第三：`Dust_SentinelRobot_L_Game`

因为 `tflm` 的结构更适合继续接入新的板级支持、算法模块、消息机制、构建裁剪和推理能力。

---

## 五、如果把三者看成我的成长轨迹

我觉得这三者不是彼此否定，而是很自然的演进。

### 第一阶段：先学会做“可复用框架”

`basic_framework-master` 让我看到了：

- 嵌入式项目不能只写功能
- 需要层次
- 需要规则
- 需要文档
- 需要为别人考虑

### 第二阶段：再学会做“完整整机工程”

`Dust_SentinelRobot_L_Game` 更像把这些理念真正落在一台机器人上。

这一步最重要的不是抽象，而是把：

- 通信
- 控制
- 任务
- 模块
- 整机逻辑

真正跑通。

### 第三阶段：最后开始形成“自己的架构语言”

`tflm` 代表的是：

不再只是借用现成框架思想，而是开始重新组织：

- 项目层
- 模块层
- 驱动层
- 算法层
- 话题层
- 板级配置层

这说明已经从“会用框架”逐渐走向“会设计框架”。

---

## 六、最后的结论

如果我要给三者下最简短的评价：

- `basic_framework-master`：最适合传承和复用，像一套队伍级 SDK。
- `Dust_SentinelRobot_L_Game`：最有实战气质，像一台机器人打出来的成熟工程。
- `tflm`：最体现个人思考，像正在成型的下一代框架。

如果只问“哪个最好”，这个问题其实不准确。

更准确的说法是：

- 想让很多人长期共用，`basic_framework-master` 很强。
- 想快速把一台机器人真正做出来，`Dust_SentinelRobot_L_Game` 很强。
- 想沉淀自己的方法论，并往更干净的架构方向走，`tflm` 最有前途。

所以这三者合在一起，刚好能说明一条非常清楚的成长路径：

**从借鉴成熟框架，到独立完成整机落地，再到提炼出自己的框架体系。**
